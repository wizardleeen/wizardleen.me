本文记录了我使用 Claude Code（以下简称 CC）从零开发一个类 Vercel 轻量级云平台的历程，以及过程中的思考与经验总结。

## 产品介绍

### 痛点

在国内部署前端项目一直很麻烦。以阿里云为例，搭建一个网站需要分别配置 OSS、CDN 和证书，每次部署都要上传文件并刷新缓存。而 Vercel、CloudFlare 等海外平台缺少国内 CDN 节点，访问速度慢，且时常遭遇封锁。

### 解决方案

为此，我基于阿里云打造了一个类 Vercel 的部署平台——**秒部**，实现 git push 即可一键部署。

项目地址：https://miaobu.metavm.tech

### 核心功能

- git push 一键部署
- 支持前端、Node 及 Python 后端
- CDN 全球加速
- 自动签发 HTTPS 证书
- 自定义域名绑定
- 快速版本回滚

目前平台已正式上线，并已接入多个站点。

## 开发过程

秒部完全由 CC 开发完成，全程没有手写一行代码。

### 原型搭建

第一步是让 CC 搭建基础原型，使用的 Prompt 如下：

```markdown
Build a lightweight, Vercel-like deployment platform targeting the China market, powered by Alibaba Cloud.
Core requirements:

* Deploy frontend projects with a single git push
* GitHub OAuth for user login
* Alibaba Cloud OSS + ESA for website hosting and delivery

Tech stack:

* Python backend
* Modern, clean architecture
```

### 环境搭建

代码编写完成后，我让 CC 在本地用 Docker 启动前端、后端、数据库等容器。由于容器操作无需 sudo 权限，CC 可以直接管理整个环境。

本地环境就绪后，接下来便是云环境集成。秒部需要调用阿里云 API 来操作 OSS 和 CDN，而这一环节容易出错——API 传参错误、密钥配错、Bucket ID 配错等等。手动调试很耗时，于是我给 CC 提供了一个临时的阿里云 Access Key，让它自行操作和调试云资源。

### 问题排查

当 CC 能直接与运行环境交互时，其排查问题的能力非常出色。举个例子：一次部署失败涉及 GitHub、秒部服务和阿里云三个环节，如果靠人工排查，费时费力；而 CC 快速采集各方信息后，很快就锁定了问题根源。

### 持续迭代

原型跑通后，我继续驱动 CC 实现更多功能：Python/Node 后端支持、版本回滚、UI 重构等。每轮迭代的工作流大致如下：

* 开启会话，描述需求
* 审阅 CC 的方案，确认后进入编码
* 构建本地容器，调试
* 编写 commit 信息，提交代码
* 执行数据迁移，部署上线
* 整理本次会话中沉淀的新知识并归档
* 关闭会话

CC 在绝大多数情况下都能独立完成调试，但偶尔也会陷入僵局，这可能源于环境配置差异，也可能是知识盲区。此时需要开发者主动提供线索，帮它突破瓶颈。

### 警惕项目成为黑盒

依赖 CC 写代码很容易滋生惰性：只管提需求，不关心实现细节。至少在当前阶段，这条路行不通。

AI 生成代码存在**短视倾向**：它聚焦于眼前的需求或当前的 Bug，倾向于选择最简路径，往往会导致架构劣化。比如 CC 曾在 API 接口中采用轮询方式检查服务状态——这是典型的反模式。它之所以这样做，是因为当时秒部尚未引入 Worker 模块，而在"新建一个模块"和"用简单方案先顶上"之间，CC 选择了后者。

随着项目复杂度提升，架构把控愈发关键。开发者必须理解并主导架构演进，**绝不能让项目成为黑盒**。

### 关于技术方案设计

对于简单明确的需求，直接交给 AI 出方案、开发者做审阅即可。但涉及重大系统升级时，技术方案需要更深层的思考。尤其是当方案涉及新颖的架构设计时，AI 目前还很难给出让人满意的答案。此时就需要由人来主导设计，输出技术方案文档，再交给 AI 去落地执行。

### "不写代码"或将成为常态

在代码实现、API 文档编写、commit message 生成等具体执行层面，AI 的表现已经超越绝大多数开发者。它能不厌其烦地处理边界条件、补充注释、编写单元测试，而这些恰恰是人类最容易敷衍的环节。而且 AI 产出的代码在可读性上，也明显优于大多数人手写的代码。按照这个趋势，开发者彻底告别手写代码，或许即将成为现实。

### 多线并行

使用 CC 开发时，你会发现自己有了大量空闲时间，因为大多数时候都在等 CC 完成任务。因此我经常同时推进多条工作线：一个会话在实现新功能，另一个在排查并发问题，而我自己则在构思设计下一个迭代的新特性。

## 总结

五天时间，从零到上线一个类 Vercel 平台，这在过去几乎不可想象。我们正在经历一次重大的行业变革，还没上车的小伙伴，抓紧了！
